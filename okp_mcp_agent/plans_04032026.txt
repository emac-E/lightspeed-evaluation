  Plan D: Pattern Recognition & Batch Fixing Strategy

  Goal: Maximize agent pattern recognition by grouping similar failures and fixing additively.

  Problem with Sequential Fixing:
  - Going ticket-by-ticket misses patterns across similar bugs
  - Each fix tested in isolation - expensive, slow
  - Doesn't leverage agent's ability to recognize failure types
  - Risk of conflicting fixes if tickets share root causes

  Pattern-Based Grouping Approach:

  1. Failure Analysis Phase
     Run diagnostics on ALL open tickets first:
     ↓
     Classify failures by type:
     - Retrieval failures (URL F1 < 0.7)
       → Group by missing doc patterns
       → Example: All "lifecycle/EOL" tickets together
     - Answer quality failures (MRR low, retrieval OK)
       → Group by LLM issues vs content issues
     - Mixed failures (both retrieval + answer)
       → Group by topic/intent (e.g., all networking tickets)
     ↓
     Identify root cause patterns:
     - Same missing document across 5 tickets → boost rule fix
     - Same query type failing → intent detection fix
     - Same terminology confusion → synonym fix

  2. Batch Fix Strategy (Additive Merging)

     Instead of: Fix ticket A → Test → Fix ticket B → Test

     Do: Fix tickets [A, B, C] → Merge all fixes → Test once

     Workflow:
     a) Agent creates fix branch for each ticket:
        - feat/fix-RSPEED-2480 (boost "lifecycle" queries)
        - feat/fix-RSPEED-2481 (boost "EOL" intent)
        - feat/fix-RSPEED-2482 (add "end of life" synonyms)

     b) Merge all fixes into integration branch:
        - feat/batch-eol-improvements
        - Contains all 3 fixes combined

     c) Run regression ONCE on combined changes:
        - Level 1: RHEL 10 + RHEL-specific test sets
        - Level 2: CLA gating tests

     d) If regression passes:
        - Create single PR with all fixes
        - Or: Separate PRs but mention they're part of a batch

     e) If regression fails:
        - Binary search to find problematic fix
        - Remove it, keep the good ones

  3. Two-Level Regression Strategy

     Level 1: Development Test Sets (Fast feedback)
     - RHEL 10 questions (new version coverage)
     - RHEL-specific questions (domain accuracy)
     - Purpose: Catch obvious breaks quickly
     - Run on: Every batch of fixes
     - Threshold: Must maintain or improve scores

     Level 2: CLA Gating Tests (Quality gate)
     - Full CLA test suite (critical user scenarios)
     - Negative tests (edge cases)
     - Purpose: Production-ready validation
     - Run on: Before PR creation only
     - Threshold: Zero regressions allowed

     Efficiency:
     - Don't run Level 2 until Level 1 passes
     - Save expensive gating tests for final validation
     - Iterate faster with smaller test sets

  4. Stability Analysis (Multi-Run Validation)

     Problem: Single-run scores can be misleading
     - LLM non-determinism causes score variance
     - One good run doesn't mean stable improvement
     - Need confidence that fix is robust

     Solution: Multi-run regression (N=3 or N=5)

     Before marking fix as "stable":
     - Run regression 3 times
     - Calculate mean ± std dev for each metric
     - Check for wavering:
       ✅ Stable:   Mean=0.85, StdDev=0.02 (tight distribution)
       ⚠️  Unstable: Mean=0.85, StdDev=0.15 (wide variance)

     Decision criteria:
     - If StdDev > 0.10: Fix is unstable, investigate why
     - If StdDev < 0.05: Fix is stable, proceed
     - Compare variance before/after fix:
       → Fix should reduce variance, not increase it

     When to run:
     - On final integration branch before PR
     - Not on every iteration (too expensive)
     - Use for critical fixes or controversial changes

  5. Pattern Recognition Implementation

     Add ticket clustering step before fix loop:

     def cluster_tickets_by_failure_pattern(tickets: List[str]) -> Dict[str, List[str]]:
         """Group tickets by failure pattern for batch fixing.

         Returns:
             {
                 'retrieval_lifecycle': ['RSPEED-2480', 'RSPEED-2481'],
                 'retrieval_networking': ['RSPEED-2113', 'RSPEED-2123'],
                 'answer_quality_eol': ['RSPEED-2482', 'RSPEED-2745'],
                 ...
             }
         """

     For each cluster:
     - Agent identifies common pattern
     - Proposes unified fix approach
     - Creates fixes for all tickets in cluster
     - Tests additively (merge all, test once)

     Efficiency gains:
     - 10 tickets, 3 patterns → 3 test runs instead of 10
     - Agent learns pattern once, applies to all
     - Better root cause fixes vs. band-aids

  6. Integration with Existing Workflow

     Modified fix_ticket_with_iteration():

     # BEFORE (sequential):
     for ticket in tickets:
         fix_ticket(ticket)
         test_ticket(ticket)
         if passed: commit_ticket(ticket)

     # AFTER (batch + additive):
     clusters = cluster_tickets_by_failure_pattern(tickets)

     for pattern, ticket_group in clusters.items():
         # Fix all tickets in cluster
         fix_branches = []
         for ticket in ticket_group:
             branch = fix_ticket(ticket)
             fix_branches.append(branch)

         # Merge all fixes additively
         integration_branch = merge_fixes(fix_branches)

         # Test ONCE on combined changes
         # Level 1: Fast regression
         if not test_rhel_sets(integration_branch):
             # Binary search to find bad fix
             bad_fix = bisect_failures(fix_branches)
             remove_fix(bad_fix)
             continue

         # Level 2: Gating tests (only if Level 1 passed)
         if not test_cla_gating(integration_branch):
             bad_fix = bisect_failures(fix_branches)
             remove_fix(bad_fix)
             continue

         # Stability check (multi-run)
         stability = run_stability_check(integration_branch, n_runs=3)
         if stability['std_dev'] > 0.10:
             print(f"⚠️  Unstable fix: {stability}")
             # Investigate or mark for manual review

         # All checks passed - ready for PR
         create_pr(integration_branch, ticket_group)

  Key Benefits:
  - Fewer test runs (3 patterns vs 10 tickets)
  - Better fixes (root cause vs symptoms)
  - Faster iteration (batch testing)
  - Confidence in stability (multi-run validation)
  - Scales better (100 tickets → ~10-20 patterns)

  Implementation:

  def generate_escalation_report(
      self,
      ticket_id: str,
      iterations: List[IterationResult],
      final_score: float,
      escalation_reason: str
  ) -> str:
      """Generate comprehensive escalation report for human review.

      Args:
          ticket_id: Ticket being escalated
          iterations: All iteration results with reasoning, changes, scores
          final_score: Best score achieved
          escalation_reason: Why agent gave up

      Returns:
          Markdown-formatted report with all context
      """

      report = []

      # Header
      report.append("=" * 67)
      report.append(f"🚨 ESCALATION REPORT: {ticket_id}")
      report.append("=" * 67)

      # Summary
      report.append(f"\nStatus: REQUIRES HUMAN INTERVENTION")
      report.append(f"Escalation Reason: {escalation_reason}")
      report.append(f"Attempted: {len(iterations)} iterations")
      report.append(f"Final Score: {final_score:.2f}")

      # Iteration history
      for i, iteration in enumerate(iterations, 1):
          report.append(f"\nIteration {i}/{len(iterations)} - Model: {iteration.model}")
          report.append("-" * 53)
          report.append(f"Strategy: {iteration.strategy}")
          report.append(f"\nAgent Reasoning:\n{iteration.reasoning}")
          report.append(f"\nChanges Applied:\n{iteration.changes}")
          report.append(f"\nResults:")
          report.append(f"  Composite: {iteration.score_before:.2f} → {iteration.score_after:.2f}")
          report.append(f"\nAnalysis: {iteration.analysis}")

      # Agent assessment
      report.append("\nAGENT ASSESSMENT")
      report.append("-" * 67)
      report.append(f"Root Cause: {self.diagnose_root_cause(iterations)}")
      report.append(f"Why Agent Cannot Solve: {self.explain_limitations(iterations)}")

      # Recommendations
      report.append("\nRECOMMENDED HUMAN ACTIONS")
      report.append("-" * 67)
      report.append(self.generate_recommendations(ticket_id, iterations))

      return "\n".join(report)
